The Challenge

Slow loading times appear even slower when using spinner-like loading design. There is no difference between loading and an error. Loading pattern is handled as a uniform form of loading regardless of context.

Loading charts on dashboard is done on the user's computer which can't really be a foreseen factor on more demanding graphs.

Psychology of loading times

The time spent on a loading animation or pattern should reflect the perceived complexity of a task: in the context of Zurvey.io, generating a dashboard would be a difficult task, therefore longer waiting times are appropriate.

However, loading the Home screen should feel snappy and effortless since essentially this is a list view. Most of the loading should happen on the dashboard and not here.

Beyond 1 second

When an expected loading time is longer than 1 second, that's when anything should be shown. For really long and complex tasks there are a few options.

Suggestions

  • Skeleton UI for list views, settings and dataset creators

Conclusion

Alternative visualisation for loading is a step to the right direction, however the actual loading times are more important. The animations will ease the sense of slow loading times, however, it can look impressive, it is not an be-all and end-all addition to the experience enhancement.
